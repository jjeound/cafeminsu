# Architecture Decision Records

## 철학
MVP 속도 우선. 외부 의존성 최소화. **작동하는 최소 구현**을 택하되, 디자인 토큰 일관성과
테스트 가능성은 타협하지 않는다. 어두운 곳에 숨은 복잡함보다 단순하고 교체 가능한 구조를 선호한다.

---

### ADR-001: UI는 Jetpack Compose
**결정**: 화면 전체를 Jetpack Compose로 구현 (XML View 시스템 미사용).
**이유**: 선언형 UI로 디자인 토큰(테마)·상태 바인딩이 직관적. 단일 액티비티 + Composable 네비게이션.
**트레이드오프**: 일부 서드파티는 여전히 View 기반 → 필요 시 `AndroidView`로 래핑.

### ADR-002: MVVM + 단방향 데이터 흐름(UDF)
**결정**: ViewModel이 `StateFlow<UiState>`를 노출하고 Composable은 상태를 받아 렌더, 이벤트는 위로.
**이유**: 풀 MVI 대비 보일러플레이트가 적고 Compose 친화적. 테스트(Turbine)와 잘 맞음.
**트레이드오프**: 엄격한 인텐트 단일화는 포기. 대신 화면별 sealed UiState로 일관성 유지.

### ADR-003: 디자인 토큰 = 단일 진실, hex 금지
**결정**: 모든 색·치수·타이포는 `docs/DESIGN_SYSTEM.md` 토큰으로만 참조. hex는 `ui/theme/Color.kt`에만.
**이유**: Figma Variable ↔ Compose Theme 1:1 유지, AI 슬롭/임의 색 난립 방지.
**트레이드오프**: 초기 테마 셋업 비용. 대신 이후 화면 구현이 빨라지고 일관됨.

### ADR-004: 음성 주문 — STT + 주문 파싱 분리
**결정**: M-04는 (1) STT로 텍스트화 → (2) 파싱 UseCase로 `{메뉴,옵션,수량}` 구조화 2단계.
파싱은 **인터페이스로 추상화**해 규칙기반 ↔ LLM 구현을 교체 가능하게 둔다.
**이유**: STT와 의미 파싱의 관심사 분리. MVP는 온디바이스 `SpeechRecognizer` + 단순 파서로 시작 가능.
**트레이드오프**: 고급 자연어 이해는 LLM 연동(키·비용) 전까지 제한. 키 필요 시 해당 step `blocked`.

### ADR-005: 키오스크는 MVP 제외
**결정**: 1차 릴리스는 모바일 앱만. 키오스크는 후속 트랙.
**이유**: 범위 축소로 MVP 속도 확보. 디자인 토큰은 플랫폼 비종속이라 재사용 가능.
**트레이드오프**: 키오스크 전용 UX(대화면·접근성) 검증은 연기.

### ADR-006: 백엔드는 Mock/Local 우선
**결정**: 메뉴·주문·적립·결제는 Repository 인터페이스 뒤에 두고 MVP는 Mock/Room 구현으로 시작.
실제 서버/PG는 DI 모듈 교체로 붙인다.
**이유**: 백엔드·PG 계약 미확정 상태에서도 UI/플로우를 끝까지 빌드·검증 가능.
**트레이드오프**: 실제 결제·동기화는 후속. 실 키가 필요한 결제 step은 `blocked` 처리.

### ADR-007: 빌드 설정은 Kotlin DSL + 버전 카탈로그
**결정**: `build.gradle.kts` + `gradle/libs.versions.toml`로 의존성·버전 일원화.
**이유**: 타입 안전, 의존성 중앙 관리, 멀티모듈 확장 대비.
**트레이드오프**: Groovy 대비 일부 예제 변환 필요.

### ADR-008: 토큰/세션 저장 = EncryptedDataStore
**결정**: 인증 토큰·세션은 EncryptedDataStore(또는 Keystore 래핑)에 저장, 평문 금지. 로그아웃 시 와이프.
**이유**: 단말 탈취·루팅 시 토큰 유출 방지. Room은 도메인 캐시 전용으로 분리.
**트레이드오프**: 암호화 I/O 비용·키 관리 복잡도. 보안상 수용.

### ADR-009: 네트워크는 HTTPS 강제 · cleartext 차단
**결정**: 모든 통신 HTTPS, `usesCleartextTraffic=false` + networkSecurityConfig. 인증서 핀닝은 추후 옵션.
**이유**: 중간자 공격·평문 노출 차단.
**트레이드오프**: 로컬 개발 시 평문 엔드포인트 별도 설정 필요.

### ADR-010: 결제 안전 — PAN 미저장 · PG 토큰화 · 멱등키
**결정**: 카드 PAN/CVC 미저장·미로깅, PG 토큰만 사용. 결제는 멱등키로 중복 방지.
**이유**: PCI 범위 축소, 중복 결제·민감정보 유출 방지.
**트레이드오프**: PG 토큰화 연동 의존. 실 키 필요 step은 `blocked`.

### ADR-011: 표준 에러 모델 AppResult / DomainError
**결정**: 외부 호출 결과는 `AppResult<T>`로, 오류는 `DomainError`로 통일(예외 전파 금지).
**이유**: 화면별 일관된 에러 처리, 테스트 용이, 자율 빌드 step 간 일관성.
**트레이드오프**: 래핑 보일러플레이트. 공통 헬퍼로 완화.

### ADR-012: 점주(매장 관리자) 모드 도입
**결정**: 점주 **기본 운영 화면**(로그인·대시보드·실시간 주문·메뉴·매출)을 본 트랙에 포함한다. 진입은 `LOGIN` 화면
하단 **"점주 로그인 →"** 링크. 인증은 고객(카카오 OAuth)과 분리된 **아이디/비밀번호**(백엔드 전까지 Mock).
세션에 역할(`CUSTOMER`/`OWNER`)을 두고, 점주는 전용 라우트 + 하단 탭(대시보드·주문·메뉴·매출)으로 분리한다.
**이유**: 매장이 직접 주문을 접수·관리하고 품절/매출을 운영해야 앱이 닫힌 루프로 동작. ADR-005(키오스크 제외)와
달리 모바일 점주 콘솔은 동일 디자인 토큰/컴포넌트로 저비용 재사용 가능. 고급 운영(자동 정산·멀티매장·직원 권한)은 후속.
**트레이드오프**: 역할 분기로 네비/세션 복잡도 증가. 점주 화면은 고객과 토큰·컴포넌트를 공유해 비용 최소화하고,
실 점주 인증·주문 푸시 연동은 백엔드 수급 후(키 필요 step은 `blocked`).
